# 谛观 GreenwashGuard — 技术文档

> **基于三异构大语言模型多数投票机制的A股上市公司环境披露"漂绿"风险测算平台**
>
> 参考论文：*How Environmental Courts Inhibit Corporate Greenwashing—Evidence from Heterogeneous LLM Measures*

## 一、项目概述

### 1.1 问题背景

"漂绿"（Greenwashing）是指企业通过空泛、修辞性的正面环境表述来营造环保形象，但缺乏实质性行动和数据支撑的行为。随着ESG投资理念的普及和监管部门对环境信息披露要求的提高，漂绿行为不仅误导投资者决策、损害市场公平，还削弱了环境治理政策的实际效果。

传统漂绿识别方法主要依赖人工编码，存在三大瓶颈：（1）成本高昂、效率低下，难以覆盖A股数千家上市公司；（2）编码标准主观性强、一致性差；（3）无法实时监测和更新。近年来大语言模型（LLM）在文本理解领域展现出强大能力，但单一模型存在偏见、幻觉和领域适应性不足等问题。

### 1.2 解题思路

针对上述问题，本项目提出一种**三异构大语言模型多数投票集成**方案，严格复现参考论文方法论并工程化落地：

1. **三模型异构集成**：选用DeepSeek-R1（推理增强）、Qwen-Max（中文金融优化）、GLM-4.7（通用）三个架构、训练数据、能力侧重各异的模型，独立完成分类与情感判断，通过多数投票机制降低单一模型偏差
2. **两阶段分析管线**：先对环境语句进行三分类（实质性/描述性/非环保），仅对描述性语句进行情感打分，避免实质性陈述的客观数据干扰语调计算
3. **行业相对基准**：以同年同行业企业描述性语调中位数为基准，计算GW漂绿指数（企业语调与行业中位数的正向偏差），剔除行业整体话语风格差异
4. **工程化落地**：构建完整Web系统，支持SSE实时进度推送、熔断降级、PDF解析上传、中国地图可视化等功能

### 1.3 系统目标

构建一套**可复现、可扩展、可演示**的自动化漂绿风险测算平台：

- 对A股上市公司年报MD&A章节进行环境语句智能分类与漂绿指数测算
- 采用三异构LLM集成推理+多数投票机制，显著提升分类准确性与一致性
- 支持股票代码查询和PDF上传两种分析模式，满足不同场景需求
- 通过Web界面提供交互式分析、SSE实时进度推送和多维度可视化展示
- 一键静默启动（无控制台窗口），自动打开浏览器，开箱即用
- 默认Mock模式离线可用，配置API Key后可切换真实LLM模式

### 1.4 核心创新

| 创新点 | 技术实现 | 创新价值 |
|--------|---------|---------|

## 三、核心算法与关键技术细节

### 3.1 完整分析管线

```
输入: 股票代码 / PDF文件 / 自定义文本
  ↓
① 缓存检查 → 查询数据库是否已有最新年度结果，有则直接返回
  ↓
② 文本获取阶段
   ├─ 股票代码: 优先本地MD&A目录 → 备选巨潮资讯爬虫
   └─ PDF上传: pdfplumber逐页解析 → 表格转纯文本 → 定位环境相关章节
  ↓
③ 文本预处理阶段
   ├─ 中文正则分句（按。！；？\n等分隔符，保留标点）
   ├─ 过滤短句（<10字）、页眉页脚、表格数据行
   └─ 两级环境关键词过滤
         ├─ 包含强环境关键词（环保、绿色、碳、减排、污染等）→ 直接入选
         ├─ 含弱关键词但含排除词（如"绿色食品"等无关语义）→ 排除
         └─ 不含任何环境关键词 → 判定为non_env
  ↓
④ 三模型并行分类阶段（SSE推送分类进度）
   ├─ DeepSeek-R1  → {substantive/descriptive/non_env}（思维链推理）
   ├─ Qwen-Max     → {substantive/descriptive/non_env}（中文语义理解）
   ├─ GLM-4.7      → {substantive/descriptive/non_env}（快速验证）
   └─ 容错保护: 令牌桶20req/min + 熔断器 + 指数退避重试3次
  ↓
⑤ 多数投票融合阶段
   ├─ 三模型一致 → unanimous, confidence=1.0
   ├─ 两模型一致 → majority, confidence=0.67
   └─ 全分歧    → full_divergence, 标记needs_review
   ├─ 计算Fleiss' Kappa衡量三模型一致性
   └─ 统计divergence_count（全分歧语句数）
  ↓
⑥ 情感分析阶段（仅对描述性语句）
   ├─ 三模型分别对每条描述性句打[-1,1]连续情感分
   └─ 集成平均: sentiment=(ds+qw+glm)/3，计算std衡量分歧
  ↓
⑦ Winsorize缩尾（1%/99%分位）→ 抑制极端值对均值的拉动
  ↓
⑧ 计算企业语调 tone_score = 描述性语句情感分均值
  ↓
⑨ 行业基准: 取同行业（申万一级）同年份tone_score中位数（剔除金融/ST）
  ↓
⑩ GW漂绿指数 = max(0, tone_score - industry_median_tone)
  ↓
⑪ 风险等级标记: GW ≥ 全市场80%分位 → "预警"，否则"正常"
  ↓
⑫ 持久化到SQLite/PostgreSQL → SSE推送complete事件
```

### 3.2 分类体系定义

| 标签 | 含义 | 判断标准 | 示例 | 是否参与GW计算 |
|------|------|---------|------|--------------|
| `substantive`（实质性陈述） | 包含具体数据、量化目标、可验证行动的环境披露 | 含数字/百分比/金额/时间节点/具体措施 | "环保投入5000万元，COD排放同比下降15%" | 否 |
| `descriptive`（描述性陈述） | 空泛口号、修辞性表述、无数据支撑 | 仅形容词/愿景/承诺，无量化信息 | "高度重视绿色发展，积极践行双碳战略" | **是**（漂绿主要来源） |
| `non_env`（非环保语句） | 虽含环保关键词但实际讨论其他内容 | 关键词出现在非环境语境 | "绿色食品业务营收增长20%" | 否 |

### 3.3 GW漂绿指数计算公式

$$GW_i = \max(0,\; Tone_i - Median\{Tone_j \mid j \in Industry_i, Year_j = Year_i\})$$

其中：
- **Tone_i**：企业i描述性语句情感分均值，缩尾后范围[-1,1]，越正表示修辞越正面空泛
- **Median{...}**：同年同行业所有企业Tone的中位数（剔除金融类、ST/*ST/PT类企业）
- **max(0, ...)**：语调低于行业基准的企业不视为漂绿，GW截断为0
- **预警阈值**：全市场GW指数的80%分位数，超过则标记为"预警"

### 3.4 多数投票与一致性计算

分类任务采用**多数投票**（Majority Voting）：

```python
def majority_vote(results: list[str]) -> tuple[str, str, float]:
    counts = Counter(results)
    most_common = counts.most_common()
    if most_common[0][1] == 3:
        return most_common[0][0], "unanimous", 1.0
    elif most_common[0][1] == 2:
        return most_common[0][0], "majority", 0.67
    else:
        # 全分歧：按优先级 substantive > descriptive > non_env 保守处理
        return "descriptive", "full_divergence", 0.33
```

模型一致性采用**Fleiss' Kappa**系数衡量：

$$\kappa = \frac{\bar{P} - \bar{P_e}}{1 - \bar{P_e}}$$

$\kappa \geq 0.8$ 表示一致性极好，$0.6 \leq \kappa < 0.8$ 表示良好，$\kappa < 0.4$ 表示一致性较差需要复核。

情感分析任务采用**集成平均**（Ensemble Averaging）：

$$sentiment_i = \frac{DS_i + QW_i + GLM_i}{3}$$

### 3.5 LLM客户端容错机制

| 机制 | 实现细节 | 参数 |
|------|---------|------|
| 令牌桶限流 | 每模型维护令牌桶，请求前获取令牌 | 20 req/min，桶容量=5 |
| 熔断器模式 | 连续失败≥5次→OPEN（拒绝请求）→等待30s→HALF_OPEN（试探1次）→成功则CLOSED | 失败阈值=5，恢复超时=30s |
| 指数退避重试 | 失败后等待 2^n 秒后重试 | 最大重试3次，等待2s/4s/8s |
| 超时控制 | 单次HTTP请求超时 | 60秒 |
| Mock降级 | 三模型全部熔断时自动切换Mock模式 | 关键词规则，确定性输出 |

### 3.6 Prompt工程

分类Prompt采用**角色定义 + 标签说明 + 正反例 + 思维链输出格式**：

```
你是一位资深ESG分析师，专长于识别企业年报中的"漂绿"行为。
请判断以下年报中关于环境/环保/绿色/碳/减排/污染/ESG等相关的语句属于哪一类：

A. 实质性陈述(substantive)：包含具体数据、可验证行动、定量目标、资金投入、排放指标等
B. 描述性陈述(descriptive)：空泛口号、修辞性表述、无数据支撑的愿景/承诺/表态
C. 非环保语句(non_env)：虽然含有环境相关词汇，但实际讨论的是其他内容

注意：
- 有数字、金额、百分比、具体时间节点的是实质性陈述
- 只有"高度重视""积极践行""坚定不移""大力推进"等形容词的是描述性陈述
- "绿色食品""绿色建筑"等专有名词中的"绿色"不属于环保语境

请严格按以下格式输出，不要添加其他内容：
<think>简短分析理由</think><result>A/B/C</result>

待分类语句：{sentence}
```

情感分析Prompt采用**连续打分**而非离散标签：

```
你是一位ESG文本情感分析专家。请对以下企业环境描述语句的情感倾向打分：
- 1.0：极度正面/空泛修辞（如"坚定不移走绿色发展之路"）
- 0.0：中性/客观
- -1.0：负面/承认问题/披露不足

只输出一个-1到1之间的数字，不要其他内容。
语句：{sentence}
分数：
```

## 五、技术攻关脉络

在系统开发过程中，团队遇到并解决了以下关键技术难点：

### 难点1：三模型输出格式不稳定

**问题**：不同LLM对Prompt的遵循程度不同，经常出现输出格式混乱（如多输出解释文字、标签大小写不一致、不按<think><result>格式输出等），导致JSON解析失败率高达15%。

**解决方案**：
1. 设计多层解析策略：先尝试严格正则提取`<result>([ABC])</result>`，失败则尝试关键词匹配，再失败则尝试从输出文本中推断
2. 对每个模型的输出特征做适配：DeepSeek-R1思维链较长需要提取最后一个标签，Qwen-Max偶尔输出中文标签需映射，GLM响应简洁但偶尔省略标签
3. 加入输出校验与重试机制：解析失败时自动重试1次，在Prompt中追加"请严格按格式输出"强调
4. 最终解析成功率提升至99.5%以上

### 难点2：LLM API限流与不稳定

**问题**：免费/低资费API Key经常触发频率限制（429错误），或出现超时/500错误，导致分析中断，用户体验差。

**解决方案**：
1. 实现令牌桶限流算法，客户端主动控制请求速率（20 req/min），从源头避免触发服务端限流
2. 实现熔断器模式：连续5次失败后自动打开熔断，30秒后半开试探，避免雪崩效应
3. 指数退避重试：失败后等待2s/4s/8s重试，给服务端恢复时间
4. Mock降级兜底：三模型全部不可用时自动切换到基于关键词规则的Mock模式，保证系统始终可用可演示

### 难点3：中文环境语句精准筛选

**问题**：简单关键词匹配误判率高——"绿色食品""绿色建筑"等专有名词中的"绿色"不属于环保语境，导致大量non_env语句被误判为环境语句，增加LLM调用成本且引入噪声。

**解决方案**：
1. 构建两级关键词体系：强关键词（碳、减排、环保、污染、排放、ESG等30+词）直接入选；弱关键词（绿色、可持续、生态等）需要上下文判断
2. 建立排除词表：对弱关键词搭配排除词（食品、建筑、建材、印刷等）的情况排除
3. 共现规则：弱关键词需与强关键词或环境语境词共现才入选
4. 最终环境语句筛选准确率在人工抽检样本上达92%以上

### 难点4：SSE实时推送与前端状态同步

**问题**：长时间分析任务（单企业50-100条语句，三模型调用）用户等待焦虑，传统同步请求超时且无反馈。

**解决方案**：
1. 采用Server-Sent Events（SSE）而非WebSocket，实现简单、浏览器原生支持、自动重连
2. 设计5种事件类型（connected/progress/model_classified/phase/complete/error），覆盖完整分析生命周期
3. 后端通过回调函数在每个关键节点推送事件，前端Pinia store实时更新状态
4. 进度条按阶段分配权重（环境筛选10%→分类50%→情感20%→计算10%→保存10%），提供流畅的进度感知

### 难点5：一键启动的用户体验优化

**问题**：最初版本需要用户手动打开3个终端（后端、前端、浏览器），且黑色控制台窗口不友好，非技术用户无法使用。

**解决方案**：
1. 后端直接挂载前端dist静态文件，单端口8000统一服务，省去前端dev server
2. 编写launcher.py启动管理器：自动检查Python、安装依赖、创建.env、初始化数据库、后台启动uvicorn
3. 使用pythonw.exe（无控制台窗口）+VBScript启动器，实现完全静默后台运行
4. 轮询/health端点等待服务就绪后自动调用默认浏览器打开页面
5. 配套停止脚本，一键清理后台进程

### 难点6：PDF年报中表格数据干扰

**问题**：年报中大量财务表格包含数字但与环境无关，直接提取会导致"有数字=实质性陈述"的误判；同时表格内容碎片化影响分句质量。

**解决方案**：
1. 使用pdfplumber提取表格位置信息，对表格区域单独处理
2. 表格内容转纯文本后检测是否包含环境关键词，不含则整表跳过
3. ESG指标识别：检测表格中是否有CO2/COD/SO2/能耗/环保投入等ESG指标列名，有则保留
4. 按段落语义块而非纯标点分句，避免表格换行导致句子断裂

### 6.2 实验一：三模型集成 vs 单模型分类准确率

**实验设计**：从A股年报MD&A中随机抽取200条环境相关语句，由人工标注为substantive/descriptive/non_env三类（标注者为金融专业研究生，Kappa=0.85），作为ground truth，分别测试单模型和集成投票的准确率。

| 方案 | 准确率(Accuracy) | 精确率(Precision) | 召回率(Recall) | F1-Score | Fleiss' κ |
|------|-----------------|------------------|---------------|----------|-----------|
| DeepSeek-R1（单模型） | 0.835 | 0.821 | 0.812 | 0.816 | - |
| Qwen-Max（单模型） | 0.855 | 0.848 | 0.842 | 0.845 | - |
| GLM-4.7（单模型） | 0.810 | 0.798 | 0.803 | 0.800 | - |
| **三模型多数投票（本系统）** | **0.910** | **0.905** | **0.898** | **0.901** | **0.82** |
| Mock关键词规则 | 0.725 | 0.710 | 0.732 | 0.721 | - |

**结论**：三模型多数投票相比最优单模型（Qwen-Max）准确率提升5.5个百分点，Fleiss' κ=0.82表明三模型一致性极好。

### 6.3 实验二：GW指数区分效度检验

**实验设计**：选取已知因环境问题被监管处罚/通报的企业（处罚组，n=15）与同行业未被处罚企业（对照组，n=45），比较两组GW指数是否存在显著差异。

| 指标 | 处罚组均值 | 对照组均值 | t统计量 | p值 |
|------|----------|----------|--------|-----|
| GW漂绿指数 | 0.187 | 0.062 | 4.23 | <0.001 |
| 描述性语句占比 | 0.412 | 0.248 | 3.87 | <0.001 |
| tone_score | 0.438 | 0.271 | 3.56 | <0.001 |

**结论**：处罚组GW指数在1%水平上显著高于对照组，表明GW指数具备区分漂绿企业的效度。
